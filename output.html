Hadoop Introduction
Hadoop
● HADOOP is an open-source software framework for distributed
storage and distributed processing of big datasets.
○ Latest version 3.3.4 (08/08/2022), see https://hadoop.apache.org.
● Hadoop is designed to run on clusters built from normal, commodity
hardware (and not on specialised, expensive machines).
○ Hence, Hadoop is designed with the fundamental assumption that
hardware failures are common, and should be handled gracefully.
“Pure” Hadoop Stack
A “pure” Hadoop stack consists of four components:
● Hadoop common: set of shared programming libraries used by the
other modules. We do not discuss this any further.
● HDFS, the Hadoop Distributed File System: Java-based file system to
store data across multiple machines.
● MapReduce: a programming model to process large sets of data in
parallel.
● YARN: Yet Another Resource Negotiator: handles the management
and scheduling of resource requests in a distributed environment.
Hadoop Distributed File System
What is a File System?
● File systems determine how data is stored and retrieved. ● A file system typically keep track of the following items:
○ Files: where the data we want to store are. ○ Directories: group files together in a hierarchical structure. ○ Metadata: such as file length, permissions and file types.
● The primary job of a file system is to make sure that the data is
always accessible and correct. ● Most modern file systems use a log in order to maintain consistency. ● Examples of file systems: EXT4, NTFS, …
HDFS
● HDFS (Hadoop Distributed File System) is a distributed file system
used to store data across a cluster of commodity machines.
○ Focus is on storing “very large” files with streaming data access
patterns.
● HDFS lets users connects to nodes over which the data files are
distributed as if it were one seamless file system.
○ Hence, it looks more or less as if you were working with files stored on
your local hard drive.
● HDFS puts a lot of emphasis on fault tolerance.
HDFS Design Principles
● “Very large” files: files that are hundreds of megabytes, gigabytes or
even terabytes in size. ● Streaming data access: Hadoop is built around the idea that the
most efficient data processing pattern is write-once,
read-many-times.
⟹ The assumption is that each analysis will involve a large
proportion (if not all) of the dataset.
○ So the time to read the entire dataset is more important than the
latency in reading the first record.
Anti-Patterns for HDFS
Do not use HDFS when you
● want low-latency data access (tens of milliseconds range). ● need to store lots of small files.
○ NameNode (see below) keeps all metadata in memory, about 150 bytes
is needed for each file, directory and block.
● have multiple writers or arbitrary file modifications.
○ Files in HDFS are always written by a single writer.
○ Writes are always made to the end of the file in an append-only fashion.
HDFS Components: NameNode
The NAmENODE is a single server with the following functionality
● Manages the filesystem namespace.
○ Maintains the filesystem tree and the metadata for all the files and
directories in the tree. ○ This information is stored persistently on local disk in two files: the
namespace image and the edit log.
● Maps data blocks (i.e. parts of files) to DataNodes (see below).
○ Note: block locations are not stored persistently, since this information
is reconstructed from DataNodes when the system starts.
● Manages incoming file system requests such as opening, closing
and renaming of files and directories.
HDFS Components: DataNodes
The DATANODEs are servers with the following functionality
● Store the data blocks.
● Handle file read and write requests.
● Create, delete and replicate data blocks among their drives
according to the instructions received from the NameNode.
● Periodically report back to the NameNode with the list of blocks they
are storing using a “block report”.
Note: a HDFS instance typically has many DataNodes. They are the workhorses of the file system.
NameNode: a Single Point of Failure
● Without the NameNode, the filesystem can not be used! It is a
single point of failure.
○ if the machine running the NameNode were destroyed, there would
be no way of knowing how to reconstruct the files from the blocks on
the datanodes.
○ Remember: all the information about the filesystem tree and the
metadata for all the files is stored in two files: the namespace image
and the edit log.
NameNode: a Single Point of Failure
● Hadoop provides two ways of making the NameNode resilient to
failure:
○ Back up the files that make up the persistent state of the filesystem
metadata. Hadoop can be configured to write the metadata to
multiple filesystems in a synchronous and atomic fashion, e.g. write to
local disk and a remote NFS mount.
○ Run a SECOnDARYNAmENODE. This server does not run as a NameNode.
It periodically merges the namespace image with the edit log (to
prevent the edit log from becoming too large).
Note: the state of the SecondaryNameNode always lags that of the
primary so data loss is almost certain in the event of total failure of the
NameNode.
HDFS Blocks
● HDFS blocks have nothing to do with “blocks” on a traditional disk
drive. ● The default size for a HDFS block is 128MB, so typically reasonably
large. ● A file in HDFS is broken into block-sized chunks (except for the last
block which may be smaller) which are stored as independent units.
○ By splitting a file into blocks it can, in principle, be larger than any
single disk drive in the cluster.
● To prevent data loss, a replication factor 𝑟 is set for each file. This
means that HDFS attempts to keep the number of replicas of each
block of this file equal to 𝑟.

Copying Data to HDFS
Use the copyFromLocal subcommand to copy data into HDFS:
hadoop fs -copyFromLocal input/docs/quangle.txt \ 1 hdfs://localhost/user/tom/quangle.txt 2
Perhaps it is easier to let the system pick up the default scheme and host in the URI (specified in core-site.xml).
hadoop fs -copyFromLocal input/docs/quangle.txt /user/tom/quangle.txt 1
Or, rather than specifying an absolute path for our home directory in HDFS we could use a relative path:
hadoop fs -copyFromLocal input/docs/quangle.txt quangle.txt 1
Copying Data from HDFS
We can copy the data from HDFS back to the local file system using the subcommand copyToLocal.
hadoop fs -copyToLocal quangle.txt quangle.copy.txt 1
You should then be able to verify that the file we copied back from HDFS has the same contents as the file we originally copied to HDFS.
Directory Creation and Listing
Creating a directory in HDFS is easy, simply use the mkdir subcommand:
hadoop fs -mkdir books 1
This will create a directory called books in the current HDFS directory. Finally, asking for a listing is done using the ls subcommand. E.g. to list the contents of the current directory.
hadoop fs -ls . 1
The information returned by ls is very similar to the information returned by the Unix command ls -l. One difference is that the replication factor of files is shown. For directories a _ is shown since the concept of replication does not apply to directories as they are treated as metadata and are stored by the NameNode and not by the DataNodes.

Anatomy of a File Write: Failures
When a datanode fails while data if being written to it, a number of actions are taken in order to ensure proper writing of the file, but these all happen transparently to the client.
It may be the case that some blocks are under-replicated, but the namenode will notice this and the block will be asynchronously replicated across the cluster until the target replication factor is reached.
MapReduce
What is MapReduce?
● MAPREDuCE is a programming model for processing and
generating large data sets. ● Hadoop provides an implementation on MapReduce that can run on
top of HDFS.
○ Other implementations exist, e.g. in MongoDB even though it is
deprecated from version 5.0 onwards, see
https://docs.mongodb.com/manual/core/map-reduce/
● MapReduce was first described in the paper “MapReduce: Simplified
Data Processing on Large Clusters”, by Jeffrey Dean &amp; Sanjay
Ghemawat. (More than 30 000 citations on Google Scholar!)
Map? Reduce?
● A MapReduce program is written in a functional style:
○ A Map-function (to be specified by the programmer) processes
key-value pairs as input and produces a set of (intermediate) key-value
pairs. ○ A Reduce-function (to be specified by the programmer) merges all
intermediate values with the same intermediate key.
Benefits of MapReduce Programming
Model
● A MapReduce program is automatically parallelized and executed
on a large cluster of commodity machines.
○ You do not need experience with distributed systems to take
advantage of its resources.
○ Fault-tolerance, data distribution, load balancing, etc. are all hidden
behind a simple interface.
○ Re-execution is the primary mechanism for fault tolerance.
MapReduce Data Flow
MapReduce Data Flow
● A MapReduce job may operate on a (very) large file. Each mapper
sees a partition of the file. ● Typically, the map job will process its partition line by line. ● The mapper produces (key, value)-pairs (and they are written to local
disk). ● The MapReduce framework then shuffles this data so that all data
items with the same key (from all the mappers!) end up and the
same reducer.
○ By default this is done by hashing the keys and then bucketing the
hashes.

YARN
What is YARN?
● YARN stands for Yet Another Resource Negotiator ● YARN provides APIs for requesting and working with cluster
resources.
○ However, these APIs are typically not used by user code.
○ Instead, users write to higher-level APIs provided by the distributing
computing frameworks. These APIs typically hide the resource
management details from the user.
What is YARN?
YARN Core Services
YARN provides its core services via two types of long-running daemon
● A RESOuRCEMAnAgER (one per cluster) to manage to use of resources
across the cluster. ● NODEMAnAgERS (running on all nodes of the cluster) to launch and
monitor containers.
○ A container executes an application-specific process with a
constrained set of resources (like memory, CPU, …). A container may be
a Unix process or a Linux cgroup.
○ Note: these containers are not related in any way to Docker containers!
A YARN Application Run
A YARN Application Run
1. A client contacts the ResourceManager and asks it to run an
application master process. 2. The ResourceManager then finds a NodeManager that can launch
the application master in a container. What happens next depends
on the application.
○ It could simply run a computation in the container and return the
result to the client.
○ Or, it could request more containers from the ResourceManager, and
use them to run a distributed computation. (This is what happens
when running a MapReduce Job.)
Note: YARN itself does not provide any way for parts of the application to communicate with one another.
Resource Requests
● YARN has a flexible model for making resource requests: a request
for a set of containers can express the amount of computer
resources required for each container, as well as locality constraints
for the containers in that request. ● Locality can be used to request a container on a specific node or in a
specific rack.
○ When launching a container to process an HDFS block (e.g. to run a
map task in MapReduce), the request will ask for container on one of
the three nodes containing a replica of the block, or on one of the racks
containing the block or when all this fails, anywhere on the cluster.
YARN Scheduling
● In an ideal world, all requests that a YARN application makes would
be granted immediately. ● When the cluster is busy, however, an application will often need to
wait to have some of its requests fulfilled. ● The YARN SCHEDuLER allocates resources to applications according
to some defined policy. Three schedulers are available in YARN:
○ FIFO: First-in, First-out
○ Capacity scheduler
○ Fair scheduler
Illustration Different Schedulers
FIFO Scheduler
● Doesn’t need any configuration or explanation. FIFO scheduler
places applications in a queue and runs them in order of submission. ● Not suitable for shared clusters. Large applications will use all the
resources.
Capacity Scheduler
● A separate dedicated queue allows the small job to start as soon as it
is submitted. ● Overall cluster utilization is worse since the queue capacity is
reserved for jobs in that queue (even if there are no such jobs).
Fair Scheduler
● There is no need to reserve a set amount of capacity. ● Resources are dynamically balanced between all running jobs. ● Note: small job has to wait until some resources are freed. When the
small job finishes the large job goes back to using full cluster
capacity. ● Overall effect is both high cluster utilization and timely small job
completion.
